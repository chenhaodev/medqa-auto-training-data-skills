 Plan: /medqa Skill — Medical LLM Dataset Selector + Distillation Generator

 Context

 Medical LLM practitioners spend hours researching which datasets to use, what format to convert them to, and how to configure Unsloth for their hardware. This skill
  encodes research-backed expert knowledge into an interactive wizard triggered by /medqa. It produces a complete, runnable training setup (train.py,
 train_config.yaml, dataset_selection.md, README.md) under ./medqa-training/.

 Research Basis (recent high-cited papers, Nature/NPJ/ICML)

 - Meerkat (npj Digital Medicine, 2025): Fine-tuning 7–8B models on 441K examples extracted from 18 medical textbooks outperformed counterparts by 22.3% on six exam
 datasets. Key insight: chain-of-thought from authoritative textbooks > large noisy datasets.
 - MedCaseReasoning (Stanford/arXiv, 2025): Fine-tuning on 14K clinical case reports from PubMedCentral improved diagnostic accuracy +29% and reasoning recall +41%.
 Even top open-source models (DeepSeek-R1) only reach 48% accuracy on this set.
 - MedS-Ins (npj Digital Medicine, 2024): 58 medically-oriented corpora, 5M instances, 19K instructions, 122 tasks. Training MMedIns-Llama 3 on this outperformed all
  prior open medical LLMs across nearly all clinical tasks.
 - DPO > SFT for complex tasks (JMIR, 2025): DPO after SFT improves clinical reasoning accuracy by 7–8% for Llama3/Mistral. SFT alone is sufficient only for simple
 rule-based tasks.
 - DeepSeek-R1 / GRPO (Nature, 2025): GRPO trains by sampling a group of outputs per prompt, using group-mean reward as baseline (no separate critic). Two reward
 types: accuracy reward (correct answer = 2.0) + format reward (correct <think>...</think><answer>X</answer> tags = 1.0). Avoids reward hacking via verifiable
 rule-based rewards only — no neural reward model.
 - Med-RLVR (arXiv 2025): RLVR applied to medical QA achieves +8% out-of-distribution accuracy over SFT, comparable to SFT on in-distribution tasks. Confirms
 verifiable reward signals generalize better than imitation learning.
 - Fleming-R1 (arXiv 2025): Two-stage GRPO pipeline: (1) Chain-of-Thought cold start with SFT on medical reasoning traces, (2) GRPO-based RLVR with curriculum
 learning (easy → hard MCQ). Achieves expert-level reasoning on USMLE/MedMCQA.
 - Curriculum learning principle (2024–2025): Easy-to-hard data ordering outperforms random mixing. Medical domain: start with broad instruction-following data, then
  move to hard clinical cases.
 - HealthBench (OpenAI, 2025): 5,000 realistic health conversations graded by 262 physicians across 26 specialties — best eval benchmark for complex multi-turn
 reasoning.
 - MedXpertQA (ICML 2025): 4,460 expert-level specialty board questions spanning 17 ABMS specialties — harder than standard MedQA/USMLE benchmarks.

 Data Generation via Distillation (new addition)

 The skill also generates synthetic training data by distilling from a strong teacher model using the distilabel CLI — the CLI-native alternative to the GUI-only
 easy-dataset tool:

 - distilabel (argilla-io/distilabel): Python framework with distilabel pipeline run --config pipeline.yaml CLI. Research-paper-backed. Chains Generator → Labeler →
 Filter steps. Pushes to HuggingFace Hub. Install: pip install distilabel.
   - CLI: distilabel pipeline run --config medical_qa_pipeline.yaml
   - Teacher model options: GPT-4o, Claude 3.7 Sonnet, Qwen2.5-72B (local), DeepSeek-R1
 - easy-dataset (ConardLi/easy-dataset): 10K+ star GUI tool (Next.js, runs on port 1717). No native CLI — runs as web app via npm run start or Docker. Good for
 document chunking + QA generation but requires browser. The skill should use distilabel as the CLI path, with easy-dataset as an optional GUI alternative.
 - EasyDistill (modelscope/easydistill): Alibaba's KD toolkit with JSON-config CLI (easydistill --config config.json). More training-oriented (SFT + ranking + RL)
 than data-generation-oriented. Best for white-box KD.

 Recommended CLI path in the skill: distilabel pipeline run with a generated YAML config that:
 1. Loads user's medical documents (PDF/txt/JSONL)
 2. Chunks them into passages
 3. Generates QA pairs using teacher model (via API or local)
 4. Filters by quality score
 5. Outputs to JSONL + optionally pushes to HuggingFace Hub

 GRPO reward function design (generated in train.py for GRPO method):
 # Reward function for medical MCQ (rule-based, no neural reward model)
 def medical_reward_fn(completions, answers, **kwargs):
     rewards = []
     for completion, answer in zip(completions, answers):
         reward = 0.0

         # 1. Format reward (0 or 1.0): must have <think>...</think><answer>X</answer>
         has_think = bool(re.search(r'<think>.*?</think>', completion, re.DOTALL))
         has_answer = bool(re.search(r'<answer>[A-E]</answer>', completion))
         if has_think and has_answer:
             reward += 1.0

         # 2. Accuracy reward (0 or 2.0): correct answer letter
         match = re.search(r'<answer>([A-E])</answer>', completion)
         if match and match.group(1) == answer:
             reward += 2.0

         # 3. Reasoning quality reward (0–1.5): CoT length proxy
         think_match = re.search(r'<think>(.*?)</think>', completion, re.DOTALL)
         if think_match:
             cot_len = len(think_match.group(1).split())
             if cot_len >= 50:   reward += 0.5   # minimal reasoning
             if cot_len >= 150:  reward += 0.5   # substantial reasoning
             if cot_len >= 300:  reward += 0.5   # thorough reasoning

         rewards.append(reward)  # max total: 4.5
     return rewards

 Phase 4 — Data Quality & Preprocessing

 One batched AskUserQuestion (3 questions): format conversion (auto/keep/custom), cleaning steps (dedup/length filter/PII/none), train-eval split.

 Research-backed preprocessing decisions:
 - Curriculum ordering (Meerkat finding): if mixing datasets, order easy→hard within trainer
 - Length filter: for long-context datasets (MedCaseReasoning avg ~600 tokens, medical-o1 ~800 tokens), use max_seq_length * 0.9 as ceiling
 - CoT quality gate: for reasoning datasets, filter examples where reasoning trace < 100 tokens (degenerate)
 - Official splits always preferred: MedQA-USMLE, PubMedQA, MedCaseReasoning all have official test splits — use them for fair eval

 Format converters in references/phase-workflows.md:
 - alpaca_to_chatml(), sharegpt_to_chatml(), medqa_to_alpaca(), medcase_to_chatml()
 - MD5 deduplication on instruction field
 - Length filter: 20 tokens min, max_seq_length * 0.9 max
 - CoT quality filter: reasoning_length > 100 tokens

 Key Design Decisions (Research-Backed)

 - Difficulty tier as first-class concept (Phases 1 + 3): The research shows MedCaseReasoning hard clinical cases give far larger gains (+29% diagnostic accuracy)
 than adding more standard MCQ data.
 - Three training method tiers: SFT (simple tasks) → SFT+DPO (complex reasoning, +7–8%) → SFT+GRPO (verifiable MCQ, best OOD generalization). The skill surfaces this
  decision explicitly with research backing.
 - GRPO reward function is rule-based only (no neural reward model): Per DeepSeek-R1 (Nature 2025) — neural reward models get exploited at scale. Format + accuracy +
  CoT-length rewards are all verifiable without a separate model.
 - Two-stage GRPO (cold start then RL): Per Fleming-R1 (2025) — GRPO on a randomly initialized model diverges. Always do SFT cold start on CoT traces first, then
 switch to GRPO.
 - GRPO curriculum A→B→C: Start with easiest MCQ (MedQA-USMLE 2K) before introducing harder questions (MedXpertQA 4K). Per curriculum learning findings.
 - Curriculum ordering in train.py: for SFT, concatenate easy-first (MedQuAD/HealthCareMagic) then hard (MedCaseReasoning). Matches Phi-3/MiniCPM findings.
 - CoT quality gate: Filter reasoning traces < 100 tokens before SFT cold start — degenerate traces harm the cold start quality.
 - HealthBench as recommended eval: generated README suggests running openai/healthbench eval after training, not just measuring loss.
